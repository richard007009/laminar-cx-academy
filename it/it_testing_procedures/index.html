The IT Testing Lifecycle

Testing is not a single event that happens at the end of a project — it is a structured, continuous process that runs throughout the entire IT delivery lifecycle. Understanding where testing fits within both project delivery and operational change is the foundation of effective test management.

Testing Within the SDLC

Core Test Management Concepts

Key Definitions:

  • Test Plan: Master document defining scope, objectives, approach, resources, schedule, risks, and entry/exit criteria for a testing effort
  • Test Case: A documented set of inputs, execution conditions, and expected results that verifies a specific system requirement or behaviour
  • Test Script: Step-by-step instructions that guide a tester through executing a test case — may be manual or automated
  • Defect / Bug: Any deviation between actual and expected system behaviour discovered during testing; tracked in a defect management tool
  • Entry Criteria: Conditions that must be met before testing can begin (e.g., environment ready, build deployed, test data loaded)
  • Exit Criteria: Conditions that must be satisfied before testing is declared complete (e.g., all critical defects resolved, pass rate ≥ 95%)
  • Traceability Matrix (RTM): Maps each test case back to the business requirement it validates — ensures complete test coverage

⚠️ Common Testing Anti-Patterns to Avoid:

  • Starting test execution without a signed-off test plan or defined exit criteria
  • Testing directly in the production environment without an isolated test environment
  • Skipping regression testing after defect fixes — fixes frequently introduce new issues
  • Treating testing as a one-time phase rather than a continuous quality activity

Types of IT Testing

Different testing types serve distinct purposes and answer different questions about system quality. A comprehensive test strategy combines multiple testing types — each targeting a specific layer of the system or a specific risk area.

Testing Types Overview

Testing Type Reference

Test Type Purpose Who Performs When
Unit Testing Verify smallest code or config units work in isolation Developer / Engineer During build/development
Integration Testing Validate interfaces between components or systems QA / Test team After unit testing
System Testing (SIT) Test the complete, integrated system against requirements QA / Test team After integration testing

Test Environment and Test Data Management

A well-managed test environment is as important as the test cases themselves. Testing in an environment that does not accurately represent production will produce unreliable results and allow defects to reach live systems. Equally, using uncontrolled or real data in test environments creates serious security and compliance risks.

Environment Tier Architecture

Test Data Management

Test Data Principles:

  • Never use real production data in non-production environments without explicit data masking or anonymisation — a GDPR, HIPAA, and PCI DSS requirement
  • Data Masking: Replace sensitive values (names, SSNs, card numbers, emails) with realistic but fictitious equivalents while preserving referential integrity
  • Synthetic Data Generation: Purpose-built tools (e.g., Faker, Mockaroo, IBM Optim) generate realistic datasets with controlled edge cases and volumes
  • Boundary and Edge Case Data: Test data must include boundary values (min, max, zero, null), invalid inputs, and known edge cases — not just happy-path data
  • Data Refresh Cadence: Refresh test environment data from production (after masking) at least monthly to keep test data representative of current production state
  • Version Control: Store test data scripts and datasets in source control alongside test cases — enable reproducible test runs

🚨 Environment Contamination Risks:

  • Test environments sharing the same network segment as production — a defective test could impact live services
  • Deploying test code or test accounts to production environments — always clean up test artefacts before go-live
  • Using shared credentials across environments — each tier must have its own account set with appropriate privilege levels

Writing Test Cases and Test Plans

A well-written test case is the single most important artefact in any testing programme. Clear, precise, and repeatable test cases eliminate ambiguity, enable consistent execution by any team member, and provide an auditable record of quality validation.

Anatomy of a Good Test Case

Test Case ID Unique identifier for tracking and traceability TC-NET-0042
Requirement Ref. Links to the business/technical requirement being tested REQ-NET-012
Title Short, descriptive summary of what is being tested Verify VLAN 30 (Voice) isolation from VLAN 20 (Data)

Test Plan Structure

Standard Test Plan Sections (IEEE 829 / ISO 29119):

  • 1. Introduction & Scope: What systems, components, and requirements are in scope; what is explicitly out of scope
  • 2. Test Objectives: What the testing effort aims to prove or validate
  • 3. Test Approach: Testing types to be used, automation strategy, and tools
  • 4. Test Environment: Environment configuration, infrastructure, and access requirements
  • 5. Entry and Exit Criteria: Conditions for starting and completing each test phase
  • 6. Test Schedule and Resources: Timeline, assigned testers, and effort estimates
  • 7. Risks and Mitigations: Testing risks (e.g., environment unavailability) and contingency plans
  • 8. Defect Management: Severity classification, escalation paths, and resolution SLAs
  • 9. Sign-off: Required approvals from technical lead, QA lead, and business owner

Defect Severity Classification:

  • Critical (P1): System is completely unusable or data loss occurs — blocks go-live; must be fixed before exit
  • High (P2): Major functionality broken with no workaround available — must be fixed before exit
  • Medium (P3): Functionality impaired but workaround exists — fix in current or next release
  • Low (P4): Minor cosmetic or non-functional issue — schedule for future sprint or backlog

Infrastructure Testing — Network, Server, and Storage

Infrastructure testing validates that physical and virtual infrastructure components are correctly configured, performing to specification, and resilient to component failures before workloads are deployed. Unlike application testing, infrastructure tests often require physical access, specialised tooling, and careful sequencing to avoid service disruption.

Network Infrastructure Testing

Network Validation Checklist:

  • Physical Layer: Cable certification (TIA-568 pass/fail), fiber OTDR traces, patch panel labelling verification, and port documentation
  • VLAN Verification: Confirm correct VLAN assignment on each access port; verify trunk port allowed VLAN lists; test inter-VLAN routing and isolation
  • Routing Tests: Verify routing table entries; test path failover when a link or neighbour is removed; validate BGP/OSPF adjacencies re-form correctly
  • Redundancy/Failover: Simulate link failures (unplug cables) and verify traffic converges within defined SLA; test LACP bond failover
  • ACL and Firewall Policy: Test every permit and deny rule with actual traffic — use positive (allowed) and negative (blocked) test cases for each rule
  • DNS / DHCP: Verify name resolution for all critical hostnames; test DHCP lease assignment and scope exhaustion handling
  • Bandwidth and Latency: Run iPerf3 or BERT tests between key points; validate QoS markings and prioritisation under load

Server Infrastructure Testing

Storage Infrastructure Testing

Storage Validation Tests:

  • Connectivity: All FC HBA ports logged into fabric; all expected LUNs visible via multipath; iSCSI sessions established on both paths
  • RAID Integrity: Array management console shows all disk groups healthy; no degraded volumes or failed drives
  • Performance Baseline: Run FIO or IOMeter to establish IOPS, throughput, and latency baseline before workload deployment — compare against array specifications
  • Path Failover: Disconnect one FC/iSCSI path; verify I/O continues on the alternate path within defined RTO; reconnect and verify both paths re-establish
  • Snapshot / Clone: Create a snapshot; mount the snapshot clone to a test host; verify data integrity; confirm snapshot deletion restores capacity

Performance and Load Testing

Performance testing validates that a system meets its defined non-functional requirements (NFRs) under expected and extreme conditions. Discovering performance bottlenecks before go-live is far less costly than dealing with a performance failure under real user load in production.

Performance Testing Types

Performance Test Type Definitions

Test Type Load Profile Goal Pass Criteria Example
Baseline Test Single user / minimal load Establish performance floor reference Response <200 ms with 1 user
Load Test Expected peak concurrent users Validate NFRs under normal peak load 95th percentile <2 s at 500 users
Stress Test Beyond expected maximum Find breaking point; validate graceful degradation System fails gracefully; no data loss
Spike Test Sudden large burst then back to normal Validate elastic scaling and recovery Recovers within 60 s of spike end
Soak / Endurance Sustained load over hours/days Detect memory leaks, connection pool exhaustion No degradation after 72-hour run
Scalability Test Incrementally increasing load Identify at what point performance degrades Linear scaling up to N nodes confirmed

Common Performance Testing Tools:

  • Apache JMeter: Open-source; widely used for HTTP/HTTPS, REST API, JDBC, and FTP load testing; scriptable and extensible
  • Gatling: Code-based load testing tool with high concurrency and detailed HTML reports; popular for developer-led performance testing
  • k6: Developer-centric, JavaScript-based load testing; integrates well with CI/CD pipelines and Grafana dashboards
  • Locust: Python-based distributed load testing; easy to script complex user behaviour scenarios
  • iPerf3: Network throughput and latency testing between two endpoints — essential for infrastructure bandwidth validation
  • FIO / IOMeter: Storage I/O benchmarking — measures IOPS, throughput, and latency at the block device level

Security Testing

Security testing identifies vulnerabilities, misconfigurations, and weaknesses in systems and infrastructure before they can be exploited by malicious actors. It encompasses a range of activities from automated vulnerability scanning through manual penetration testing and code review.

Security Testing Types

Penetration Testing Phases (PTES Methodology)

Phase Activity Output
1. Pre-Engagement Define scope, rules of engagement, legal authorisation, and emergency contacts Signed Rules of Engagement (RoE) document
2. Reconnaissance Gather intelligence on targets (OSINT, DNS, WHOIS, social media) Target profile and attack surface map
3. Scanning / Enumeration Active scanning of hosts, ports, services, and banners Vulnerability list and network map
4. Exploitation Attempt to exploit identified vulnerabilities within agreed scope Proof-of-concept exploits; access gained
5. Post-Exploitation Assess impact — lateral movement, privilege escalation, data access Impact assessment and evidence
6. Reporting Document findings, severity ratings (CVSS), and remediation recommendations Executive summary + detailed technical report

⚠️ Penetration Testing Rules of Engagement — Non-Negotiable:

  • Always obtain written, signed authorisation before any penetration testing activity begins — unauthorised testing is illegal regardless of intent
  • Define explicit in-scope and out-of-scope systems — never test systems not explicitly listed in the RoE
  • Agree on testing windows and emergency stop procedures — have a named technical contact available throughout the test
  • Treat all findings as confidential — encrypt all reports and evidence; agree on data retention and destruction timelines with the client

Disaster Recovery and Failover Testing

A disaster recovery plan that has never been tested is not a plan — it is a hypothesis. DR and failover testing validates that recovery procedures work as documented, that RTO and RPO targets are actually achievable, and that staff know how to execute the plan under pressure.

DR Test Types and Increasing Scope

Failover Test Procedure — Step by Step

Infrastructure Failover Test Checklist:

  • Pre-Test Preparation: Notify all stakeholders; confirm maintenance window; snapshot/backup primary systems; assign roles (test lead, scribe, rollback owner)
  • Baseline Capture: Record current system state — service health, replication lag, active sessions, DNS TTLs
  • Initiate Failover: Execute the documented failover procedure; record start time
  • Measure RTO: Record the time each service becomes available at the DR site — compare against the defined RTO target
  • Validate Data Integrity: Verify record counts, checksums, and application data at DR site match the pre-failover baseline; measure data loss against RPO
  • Functional Smoke Test: Execute critical path smoke test cases against the DR environment to confirm application functionality
  • Failback: Reverse replication; execute documented failback procedure; verify production systems are restored to correct state
  • Post-Test Review: Document all variances from expected RTO/RPO; log defects; update runbook with lessons learned

⚠️ DR Testing Requirements:

  • DR tests must be performed at least annually — many regulatory frameworks (PCI DSS, HIPAA, ISO 27001) require documented evidence of DR test results
  • Component-level failover tests (HA, storage multipath, NIC bonding) should be performed at every infrastructure change
  • Never rely solely on tabletop exercises — they do not validate actual system behaviour, only documented procedures
  • Always define and rehearse the rollback procedure before initiating any DR test — have a tested abort path available

User Acceptance Testing (UAT)

User Acceptance Testing is the final validation gate before a system goes live. Its purpose is to confirm that the delivered solution meets the real-world business needs of the end users who will operate it — not just that it matches the technical specification. UAT defects discovered here are far cheaper to fix than post-production issues.

UAT Process Flow

UAT Best Practices

Making UAT Effective:

  • Involve Real Users Early: Include end users in test case design — they know the real-world edge cases that IT testers miss
  • Test Business Scenarios, Not Features: UAT test cases should be end-to-end business processes (e.g., "Process a purchase order from creation to payment") not individual technical features
  • Provide Training Before UAT: Users should have basic product training before UAT begins — confusion about how to use a feature is not a system defect
  • Define Clear Accept/Reject Criteria: Agree before UAT begins what pass rate and defect profile constitute UAT sign-off — prevent scope creep and endless re-testing
  • Time-Box the UAT Phase: Set a fixed UAT window; do not allow UAT to extend indefinitely — manage new requests as change items post-go-live
  • Document Sign-Off Formally: A named business owner must provide written sign-off — verbal approval is not acceptable for audit purposes

Go / No-Go Decision Criteria:

  • Zero open Critical (P1) or High (P2) defects — no exceptions
  • UAT pass rate meets or exceeds the agreed exit threshold (typically ≥ 95%)
  • All non-functional requirements (performance, security, availability) have been formally tested and signed off
  • Rollback plan documented, rehearsed, and approved — the go-live decision must always include a confirmed fallback option
  • Business sponsor has provided formal written sign-off on UAT completion

Test Automation

Test automation replaces repetitive manual test execution with scripts that run consistently, quickly, and without human error. When applied to the right tests, automation dramatically reduces regression testing effort, accelerates release cycles, and provides a safety net for continuous delivery pipelines.

Automation Pyramid — What to Automate

Automation Priority by Test Layer:

  • Unit Tests (Highest Priority): Fastest, cheapest, most stable to automate — run on every code commit; developers own these; tools: JUnit, pytest, NUnit, RSpec
  • API / Integration Tests (High Priority): Validate service interfaces and integrations; fast execution; tools: Postman, REST-assured, pytest, SoapUI
  • UI / End-to-End Tests (Selective): Automate only the most critical user journeys — UI tests are brittle and expensive to maintain; tools: Selenium, Playwright, Cypress
  • Infrastructure Tests (High Priority): Validate configuration state (Ansible idempotence tests, Terraform plan/apply validation, InSpec compliance tests)
  • Do Not Automate: Exploratory testing, usability assessments, one-off investigations, and first-time test case discovery — human judgement is irreplaceable here

CI/CD Integration — Shift-Left Testing

Automation Best Practices

Building a Sustainable Automation Suite:

  • Test Code is Production Code: Apply the same code quality standards to test scripts — version control, code review, refactoring, and documentation
  • Maintain Test Independence: Each automated test must be able to run in any order and leave no state that affects subsequent tests
  • Flaky Test Policy: A flaky test (non-deterministically passing/failing) must be immediately quarantined and fixed — flaky tests erode trust in the entire suite
  • Metrics to Track: Test execution time, pass rate trend, code coverage %, and mean time to detect (MTTD) for regressions
  • Review Regularly: Retire obsolete tests; update tests to reflect new functionality — an unmaintained test suite becomes a liability

Test Reporting, Documentation, and Compliance

Test evidence is not just an internal quality artefact — it is often a regulatory and contractual requirement. Clear, well-structured test documentation demonstrates due diligence, enables audits, supports post-incident investigations, and provides an audit trail for change management boards.

Test Report Structure

Section Content Audience
Executive Summary Overall pass/fail status, key risks, go/no-go recommendation in plain language Management, business sponsor
Test Scope and Coverage Requirements tested vs not tested; traceability matrix reference; coverage % Technical lead, QA manager

Recommendations Outstanding risks; items to address post-go-live; follow-up testing required All stakeholders
Sign-Off Section Named approval signatures from technical lead, QA lead, and business owner Governance / audit

Compliance and Regulatory Evidence Requirements

Testing Evidence Required by Key Frameworks:

  • PCI DSS (Requirement 11): Quarterly internal vulnerability scans; annual external penetration test with evidence; 6-monthly network scan after significant changes
  • ISO 27001 (Annex A.12.6): Technical vulnerability management; documented test results as evidence of control effectiveness during certification audits
  • ITIL / Change Management: Test evidence required for all significant changes before CAB approval — test plan, results, and sign-off stored in the CMDB change record
  • HIPAA / HITECH: Technical safeguard testing (access control, audit logging, encryption) must be documented and available for HHS audit
  • SOC 2 Type II: Evidence of ongoing testing activities across trust service criteria; auditors will sample test reports and defect records over the audit period
